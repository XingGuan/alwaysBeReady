> "反向代理"、"正向代理"，这两个词是从**代理服务的方向**来命名的。  

#### 一、正向代理（`Forward Proxy`）  
站在客户端（用户）的角度看：  
+ 客户端知道自己要访问谁（比如`google.com`）,但**不直接访问**，而是通过一个**中间服务器（正向代理）**去请求。   
+ 代理服务器代替客户端去访问目标服务器，然后把结果返回给客户端。
+ 正向代理**隐藏了客户端**（服务器不知道真实客户是谁）。

常见场景：  
+ 翻墙（通过代理访问国外网站）  
+ 公司内网统一出口（所有员工通过一台代理上网）
+ 缓存静态资源

```text
客户端 A ──┐
客户端 B ──┼→ 正向代理 ──→ 目标服务器（如 google.com）
客户端 C ──┘
```  
二、反向代理（Reverse Proxy）  
站在服务器（后端）的角度看：  
+ 客户端并不知道自己最终访问的是哪一台具体的服务器，它只访问**返向代理服务器的地址**。  
+ 反向代理根据规则（负载均衡、路径匹配等）把请求转发给**内部的真实服务器**，然后把响应返回给客户端。  
+ 反向代理隐藏了**真实服务器**（客户端不知道后端的集群结构）。  

常见场景：  

+ 负载均衡（把请求分发给多台服务器）
+ 统一入口（前端静态资源 + API共用一个域名）  
+ 安全隔离（不暴露后端真实端口）

```text
客户端 ──→ 反向代理（Nginx） ──┬── 服务器 1
                               ├── 服务器 2
                               └── 服务器 3
```
三、为什么叫"反向"？  
+ 正向代理：代理的对象是"客户端"，帮助客户端"向外"访问资源。  
+ 反向代理：代理的对象是"服务端"，帮助服务器"向内"接收和处理请求。  
用"方向"来区分：正向代理是客户端主动配置的"出口"代理；反向代理是服务端部署的"入口"代理，对客户端透明。

四、全栈部署中为什么用 `Nginx`做反向代理？  
+ **前端静态文件**（`HTML/CSS/JS`）和一个**后端API服务**（`Node.js` 监听 `localhost:3000`）。   
+ 如果你让浏览器`http://你的服务器IP:3000`,`CORS`问题、端口暴露、性能都不好。  
+ 用`Nginx`作为**反向代理**，让`Nginx`监听公网的`80/443`端口：  
    + 当请求路径是`/`时，`Nginx`返回静态文件(相当于充当了静态服务器)。  
    + 当请求路径是`/api/`时，`Nginx`把请求转发给`http://localhost:3000`(后端)。  
+ 对浏览器来说，它只知道在和`https://yourdomain.com`通信，完全不知道后端在3000端口、甚至不知道后端有多少个实例。    
这就是反向代理的核心作用：把内部的多台服务器、不同类型的服务，统一成一个对外的入口，对外部隐藏内部细节。 

> "反向"并不是因为"站在服务端角度"，而是因为这种代理的"请求流向"与正向代理相反。站在服务端角度只是为了方便理解它的部署位置和服务对象，但命名的真正原因是方向性对比。  

> 1.从请求流看  
> + 正向代理：客户端——>代理——>互联网（目标服务器）  
请求是从内向外发起的，代理帮助客户端出去。  
> + 反向代理：客户端——>代理——>内部服务器（真实后端）  
请求是从外向内到达代理，然后代理转向内部服务器。  
相对于正向代理的"向外"方向，这个流量是"**相反的**",所以叫**反向代理**。  

> 2.从配置主体看  
+ 正向代理需要客户端主动配置   
客户端（例如浏览器、操作系统、应用程序）必须明确设置代理服务器的地址和端口才能使用。客户端知道代理的存在，并主动将请求发送给它。   
+ 反向代理不需要客户端做任何配置   
反向代理的配置完全在服务端，由运维或开发人员在服务器上部署和设置（例如配置 `Ngnix、HAProxy`）。对客户端来说，他访问的只是反向代理暴露出来的公网域名或IP，完全不知道背后还有内网服务器集群，也感知不到代理的存在。  
因此，从配置视角也能清晰区分：  
正向代理是"客户端主动找代理"，由客户端控制出口；反向代理是"代理主动接收请求再转发"，由服务端控制入口，对客户端彻底透明。
