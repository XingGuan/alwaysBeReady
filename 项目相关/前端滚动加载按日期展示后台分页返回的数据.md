> 后端传统分页问题
 传统分页为什么"不稳定"?  
传统分页接口通常这样请求：   
```text
GET /data?page=2&size=10 // 第2页，每页10条  
```  
问题在于：数据是实时增长的。假设你第一次请求第1页，拿到了最新的10条。在请求第2页之前，有人发布了3条新内容。  
前端就算做了按天合并，也会出现**同一条数据被插入两次**，或者**某一天数据出现断层**。

> 游标分页是什么?  
换个思路：不再用"第几页"，而是跟后端说："**我最后看到的数据是这条，请给我它后面的10条**"。  
比如这样请求：  
```text
GET /data?cursor=20206-05-09T10:30:00&size=10
```    
这里`cursor`就是一个**游标**，它可以是你最后拿到那条数据的**时间戳**或**唯一递增ID**。  
后端返回时，只给游标之后的数据，并在响应里返回一个新的"nextCursor"(本页最后一条的时间)，方便你下次请求。

> 为什么推荐"基于时间/ID的游标"?  
因为你的数据本身就**按时间倒序排列**，且每一条都有日期。
用**时间戳或ID作为游标**天然契合：  
+ **时间游标**：`cursor=2026-05-09T08:30:00`,请求比这个时间更早的数据。  
+ **ID游标（如果ID是递增的）**：`cursor=12345`,请求`id<12345`的数据。  
无论期间有多少数据插入到"未来时间"，**游标这个点都不会变**。你下一把请求拿到的，永远是紧接在游标之后，比它更旧的10条，**永远不会有重复和遗漏**。  

### 和当前场景有什么关系？  
你正在做“按天分组+滚动加载”，核心痛点是：
**一页数据可能跨越多天，而且后续请求可能再次命中已渲染过的日期。**
如果用游标分页，后端保证每一页数据都是连续且无重复的，你前端只需要：
+ 拿到新数据，按天分组。
+ 与已有分组合并（同一天的数据拼在尾部，全新日期插入合适位置）。
+ 完全没有去重的烦恼。
如果后端只给传统的 `page` 分页，那么随着数据新增，你拿到的前后两页可能会有重叠，就必须在合并时做“根据ID去重”的兜底逻辑，既复杂又容易出bug。
你问的“游标”在这里指的是**分页请求时，用一个唯一标识作为“锚点”来划定下一页的范围**，而不用“第几页”这种固定数字。

### 一个直观的比喻
- **传统分页**就像你在翻一本不断被人加页的书，翻到下一页时，刚刚看的那段内容可能又被挪到更前面的位置了。
- **游标分页**就像在翻一条无限长的收据卷，你用手指掐住刚才看到的位置，说“往后给我看一截”，手指掐的位置不会因为前面又打印了新东西而改变。
所以，如果能让后端改动，就建议提供基于时间戳（或ID）的游标分页。这个改动通常很小，但在前端能把复杂的“去重合并”逻辑完全砍掉。
